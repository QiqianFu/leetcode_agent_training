  🟡 Medium（边界情况、并发风险）

  M1. Rng 一致性：只保证第一次 tool call

  seed 从 prompt hash 算，4 条同组轨迹共享 seed。但 self._rng 是单一 
  random.Random，每次 rng.choice 推进状态。
  - 若 4 条轨迹都是 "第 1 个 tool = pick_problem"，OK，同题
  - 若其中一条先 check_problem(146)（不动 rng），再 pick_problem，它的 pick 就会
   offset 一步
  - 用户要的 "组内 pick 同题" 只在 aligned 情况下成立

  Fix: 要么为每个 tool 起独立 rng（self._pick_rng, self._search_rng），要么 pick
   时用 (seed, turn_idx) 哈希，避免 rng 状态漂移。

  M2. arrow_select monkey-patch 不是线程安全

  _run_tool 里 lc.ui.arrow_select = _deterministic_select
  是全局模块属性赋值。TRL 1.1.0 sync mode 下 rollout
  是单线程顺序跑的，没问题；但如果将来走 experimental/async_grpo，这会跨 env
  互相覆盖。

  同理 os.chdir(self._tmpdir) 是进程级 cwd，也不线程安全。

  Fix（防御性）：文档里标清楚 "do not use async rollout"；或者把 tool
  实现改成纯函数式（把 workspace_root 显式传进去），彻底摆脱 cwd。

  M3. DB 连接管理细节

  _run_tool:154,203-211 的 save/restore 是对的，但每个 tool call 开关一次 SQLite
   + WAL 有开销。一条轨迹 5 次 tool call 就 5 次
  connect/close。低优先级，但累计起来影响 throughput。

  Fix（可选）：在 reset() 里 open 一次，_cleanup() 里 close。

  M4. 没有 vLLM / flash-attn

  run_grpo.sh:17 尝试装 flash-attn，失败了就用 eager。transformers 原生
  .generate 比 vLLM 慢一个量级。4 GPU × 每条 rollout 5 iter × 1024 tokens，一
  step 几分钟起步。

  Fix: 接受 throughput 慢的现实，或者再试一次 vLLM（现在 transformers 5.2+
  要求也是个大坑）。这个不是 bug，是工程妥协。
